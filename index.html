Android 7.0文件权限和应用安装

发表于 2018-02-02 | 分类于 笔记

最近要给一个应用加上提示更新功能,遇到了几个坑,虽然也很快解决了但是还是记录下来给大家当做提示教训.

Android 7.0 文件权限

代码一把梭后,在调试时就碰到了第一个问题,apk下载安装时报了 FileUriExposedException
随后了解了一下这是android 7.0新增的文件权限问题.

从Android 7.0开始,一个应用提供自身文件给其它应用使用时,如果给出一个file://格式的URI的话,应用会抛出FileUriExposedException.这是由于谷歌认为目标app可能不具有文件权限,会造成潜在的问题,所以让这一行为快速失败.

谷歌官方推荐的解决方案.即使用FileProvider来生成一个content://格式的URI

manifest声明

编写file_paths.xml

文件的名字没有要求，只要放在xml文件夹下面就可以了,所以为了保证唯一建议加上应用包名

path 表示根目录下的文件路径 .为所有
name 标记名

使用FileProvider提供的uri

然而做好了上面的操作后又报出了

查找资料后发现是自己用的第三方图片选择框架已经使用了android.support.v4.content.FileProvider

于是写个空provider继承了 FileProvider 问题解决

APK安装

终于解决权限问题后开始安装APK,心想终于解决了,然后出现了应用未安装…..

我眉头一皱发现事情并不简单versionCode和versionName我都加高了啊为什么安装会跟原应用冲突呢?

仔细一想,应该是签名问题,我当前调试用的是debug自带的签名正式版本是正式的签名.
签名后再试,果然解决了

Java中的并发编程

发表于 2018-01-22 | 分类于 笔记

并发编程的三个概念

  1. 原子性

    即一个操作或多个操作 要么全部执行并且执行过程不会被打断，要么不执行。

  2. 可见性

    指的是多个线程访问同一个变量时，一个线程修改了这个变量的值，其他线程能立即看得到

  3. 有序性

    程序执行的顺序按照代码的先后顺序执行

JAVA中内存模型

java内存模型规定所有的变量都是存在主存当中,每个线程都有自己的工作内存.线程对变量的操作都必须在工作内存中进行,不能直接对主存进行操作.并且每个线程不能访问其他线程的工作内存

1.原子性

对基本数据类型的读取和赋值操作是原子性的

Java内存模型只保证了基本的读取和赋值操作是原子性操作.
实现更大范围操作的原子性,可通过synchronized和Lock来实现

  1. 可见性

Java提供了volatile关键字来保证可见性

当一个变量被volatile关键字修饰时,它会保证修改的值会立即被更新到主存。
当有其他线程需要读取时,它会去内存中读取新值

另外通过synchronized和Lock也能保证可见性.
在执行完同步代码后释放锁之前会将对其变量的修改刷新到主存中

3.有序性

在Java内存模型中,允许编译器和处理器对指令进行重排序。但是重排序过程不会影响到单线程程序的执行，却会影响到多线程并发执行的正确性。

在Java里面，可以通过volatile关键字来保证一定的“有序性”。另外可以通过synchronized和Lock来保证有序性，很显然，synchronized和Lock保证每个时刻是有一个线程执行同步代码，相当于是让线程顺序执行同步代码，自然就保证了有序性。

深入剖析volatile关键字

  • volatile的两层语义

一旦一个共享变量(类的成员变量,类的静态成员变量)被volatile修饰后,就具备了两层语义

1) 保证了不同线程对这个变量进行操作的可见性,即一个线程修改某个变量的值,这个新值对其他线程来说是立即可见的.
2) 禁止进行指令重排

  • volatile无法保证原子性
  • volatile能在一定程度上保证有序性

由于flag变量为volatile变量，那么在进行指令重排序的过程的时候，不会将语句3放到语句1、语句2前面，也不会讲语句3放到语句4、语句5后面。但是要注意语句1和语句2的顺序、语句4和语句5的顺序是不作任何保证的。

并且volatile关键字能保证，执行到语句3时，语句1和语句2必定是执行完毕了的，且语句1和语句2的执行结果对语句3、语句4、语句5是可见的。

volatile关键字的使用场景

synchronized关键字是防止多个线程同时执行一段代码，那么就会很影响程序执行效率，而volatile关键字在某些情况下性能要优于synchronized，但是要注意volatile关键字是无法替代synchronized关键字的，因为volatile关键字无法保证操作的原子性。

通常来说，使用volatile必须具备以下2个条件：
1) 对变量的写操作不依赖于当前值
2) 该变量没有包含在具有其他变量的不变式中

内存管理笔记

发表于 2018-01-18 | 分类于 笔记

OC中的内存管理

oc作为面向对象的c,其内存也是自己管理的,但oc中对其操作进行了优化
从c的自己开辟和回收内存,改进为手动计算引用数

当引用增加时 调用retain方法为此变量引用数+1
当要释放引用时 调用release方法为引用数-1
当引用数为0时将释放内存

在xcode4后 引入了ARC模式 xcode将在编译自动时为各个对象引用数量改变时插入 retain和release方法
从此解放双手
对循环引用的解决方案为在编写代码时 一方使用强引用 一方使用弱引用

python的垃圾回收

  • 自动引用计数

    对象每增加一个引用 引用计数+1
    减少引用计数-1 当引用计数为0删除对象

    缺点:无法清理循环引用

  • 隔代回收

    python每个对象创建时 ,都会将其加入零代链表 在满足一定条件(创建对象计数-回收对象大于一定数量)后
    判断链表中对象是否有互相引用 有的引用计数-1 引用计数为0的删除 引用计数大于1且无互相引用的 加入到 一代链表

Java的垃圾回收

  • 可达性分析

    以 GC Root对象为起点向下遍历所有对象为可用对象,其余为不可用对象

    GC Root对象包括下面几种:

    1. 栈中局部变量表中引用的对象
    2. 方法区中静态属性引用的对象
    3. 方法区常量引用的对象
    4. 本地方法栈中JNI引用对象

  • 标记-清除算法

    分为标记,清除两个阶段:首先标记出所有需要回收的对象,标记完成后统一回收所有被标记的对象

    缺点:标记和清除效率都低,且产生大量不连续内存

  • 复制算法( 新生代回收算法 )

    将内存分为2小1大3块为8:1:1 ,每次只用其中一大一小,当这触发回收时,将其中存活的对象复制到另一块未使用的小内存去,然后将已经使用过的内存空间一次性清理掉

    当一些对象经过多次Minor Gc后还仍然存活就会被移动到老年代中

  • 标记-整理算法 ( 老年代回收算法 )

    让存活对象统一向一端移动(不可用对象所占内存在移动时被覆盖),然后直接清除边界以外的内存空间
